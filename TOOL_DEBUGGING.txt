### How to Check Logs

#### Primary: Console Output
When the agent runs, you should now see messages like:
```
[CRITICAL] ... 🔴 >>>>>> HANDLER CALLED FOR: get_available_slots
[CRITICAL] ... 🔴 API_CLIENT.GET_AVAILABLE_SLOTS CALLED
[CRITICAL] ... ✅ TOOL_CALL_HANDLER INVOKED
```

These markers make it crystal clear if tools are being called:
- **NOT THERE?** → Tools aren't registered or LLM isn't calling them
- **THERE BUT FAILING?** → API or connection issue (check error messages below)

#### Secondary: Log File
All logs also write to `logs/agent.log`:
```bash
tail -f logs/agent.log
```
This file will contain everything, even if console output is redirected or hidden.

### Diagnosis Workflow

#### Step 1: Verify Tool Setup (No Agent Running)
```bash
python test_tool_registration.py
```
This will:
1. Connect to your API backend
2. Authenticate 
3. List all registered tools
4. Test a `get_available_slots` call WITHOUT the agent

**Expected output:**
```
✅ APIClient connection opened
✅ APIClient authentication successful
✅ Tool definitions retrieved: 5 tools
   - get_available_slots
   - book_appointment
   ...
✅ Tool call returned: success
```

**If this fails:** Your API or credentials are wrong. Fix before testing the agent.

#### Step 2: Start Agent & Run a Test
1. Start the agent normally
2. Open the client and ask for available appointments
3. Check logs immediately:

```bash
# In another terminal
tail -f logs/agent.log | grep -i "tool\|critical\|handler"
```

**What to look for:**

✅ **TOOLS BEING CALLED (logs show these):**
```
[CRITICAL] 🔴 >>>>>> HANDLER CALLED FOR: get_available_slots
[CRITICAL] 🔴 API_CLIENT.GET_AVAILABLE_SLOTS CALLED
[CRITICAL] ✅ TOOL_CALL_HANDLER INVOKED
```
→ If you see these, tools ARE working. Problem is elsewhere (API response, agent logic, etc.)

❌ **TOOLS NOT BEING CALLED (logs DON'T show above):**
```
[INFO] 🔴 PIPECAT AGENT INITIALIZATION
[INFO] ✅ All booking tool handlers registered individually with LLM
[INFO] ✓ Registered 'get_available_slots' (cancel_on_interruption=True)
...
(no handler logs appear)
```
→ If you DON'T see handler logs, either:
- Tools aren't registered correctly
- LLM isn't choosing to call them
- The tool call is happening but logs aren't being printed

### Common Issues & Fixes

#### Issue: Logs not showing up anywhere
**Solution:**
1. Check `logs/agent.log` exists:
   ```bash
   ls -la logs/agent.log
   tail logs/agent.log
   ```
2. Ensure log level isn't set too high:
   ```bash
   echo "LOG_LEVEL=${LOG_LEVEL}" # should be DEBUG or INFO
   ```
3. Verify logs directory is writable:
   ```bash
   touch logs/test.txt && rm logs/test.txt # should succeed
   ```

#### Issue: Tool handlers registered but not called
**Possible causes:**
- LLM isn't deciding to call tools (check OpenAI API response)
- Tool schema mismatches what LLM expects
- Tools registered AFTER LLM context created

**Debug:**
1. Check tool schema is valid:
   ```bash
   python -c "from agent.tools.booking_tools import BOOKING_TOOLS_SCHEMA; import json; print(json.dumps(BOOKING_TOOLS_SCHEMA[0], indent=2))"
   ```

2. Check that context creation happens AFTER tool registration:
   ```bash
   # In logs, you should see:
   [INFO] 🔧 REGISTERING 5 TOOL HANDLERS
   [INFO] ✅ LLMContext built with 5 raw OpenAI tool schemas
   ```

#### Issue: Tool called but API returns error
**Debug:**
1. Check API response in logs:
   ```bash
   grep -A5 "API RESPONSE" logs/agent.log
   ```

2. Verify API backend is running:
   ```bash
   curl -s http://localhost:8000/api/v1/appointments/available-slots?date=2026-03-10
   ```

3. Check authentication token:
   ```bash
   grep "Auth OK" logs/agent.log
   ```
